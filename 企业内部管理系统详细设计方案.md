# 企业内部管理系统 · 后端开发设计文档（v3）

> 本文是面向后端开发的落地设计文档，在初稿基础上补齐：系统管理域的完整表设计（账号/角色/权限/日志/字典）、账号与员工的关联、数据权限模型、**完整 API 接口清单与请求/响应示例**、并发与一致性方案、认证授权流程、项目结构与工程化规范，可直接据此开展开发。
>
> **已采用的关键决策（如需调整可覆盖）：**
> - 权限模型：**RBAC（角色-权限表驱动）**，预置 5 个固定角色，同时支持后续在后台动态配置。
> - 主键：**UUID v7（有序）**，MySQL 中以 `CHAR(36)` 存储，兼顾唯一性与索引有序性。
> - 删除策略：核心业务表一律**软删除**（`deleted_at`），保证审计可追溯。
> - 时间：数据库统一存 **UTC**，展示层做时区转换；时间字段用 `DATETIME(3)`。
> - 命名：全部使用**英文蛇形命名**，字段风格统一。

## 1. 系统概述

企业内部管理系统，用于统一管理**部门、员工、工位、IT 设备**四类核心资产及其归属关系，并提供账号权限、操作审计、数据字典等系统能力。

**核心域划分：**
- 组织域：部门、员工、上下级关系
- 资产域：工位、设备及其领用/分配全生命周期
- 系统域：账号、角色权限、操作日志、数据字典
- 报表域：资产概览、闲置统计、到期预警

## 3. 角色与权限模型

### 3.1 角色定义

| 角色 code | 名称 | 数据范围 | 说明 |
|---|---|---|---|
| `super_admin` | 超级管理员 | 全局 | 全部功能与数据 |
| `hr_admin` | HR 管理员 | 全局 | 部门/员工/工位全量管理 + 报表；**无设备管理** |
| `it_admin` | IT 管理员 | 全局 | 设备全生命周期 + 工位管理；员工**只读**；报表查看 |
| `dept_manager` | 部门主管 | 本部门（含子部门，可配置） | 本部门员工/工位/设备**只读**，无编辑删除 |
| `employee` | 普通员工（自助） | 仅本人 | 只查看本人档案、工位、设备 |

> 注：原稿 `regular_admin` 更名为 `employee`，避免"普通员工"却叫 admin 的歧义。

### 3.2 权限矩阵（功能权限）

权限编码格式：`模块:操作`。✅=允许，🔒=仅只读，—=无权限。

| 权限 | super_admin | hr_admin | it_admin | dept_manager | employee |
|---|:---:|:---:|:---:|:---:|:---:|
| department:view | ✅ | ✅ | 🔒 | 🔒(本部门) | — |
| department:create/update/delete | ✅ | ✅ | — | — | — |
| employee:view | ✅ | ✅ | 🔒 | 🔒(本部门) | 🔒(本人) |
| employee:create/update/delete | ✅ | ✅ | — | — | — |
| employee:resign（离职处理） | ✅ | ✅ | — | — | — |
| employee:import/export | ✅ | ✅ | — | — | — |
| workstation:view | ✅ | ✅ | ✅ | 🔒(本部门) | 🔒(本人) |
| workstation:manage（分配/释放/编辑/维护） | ✅ | ✅ | ✅ | — | — |
| device:view | ✅ | — | ✅ | 🔒(本部门) | 🔒(本人) |
| device:manage（入库/领用/退还/维修/报废） | ✅ | — | ✅ | — | — |
| report:view | ✅ | ✅ | ✅ | 🔒(本部门) | — |
| system:user（账号管理） | ✅ | — | — | — | — |
| system:role（角色权限） | ✅ | — | — | — | — |
| system:log（操作日志） | ✅ | — | — | — | — |
| system:dict（数据字典） | ✅ | — | — | — | — |

### 3.3 数据权限（行级过滤）规则

- **全局角色**（super/hr/it）：不加数据范围过滤。
- **dept_manager**：`WHERE department_id IN (本人所在部门 + 其子部门ID集合)`。子部门通过部门树 `parent_id` 递归求得，结果缓存到 Redis。
- **employee**：`WHERE employee_id = 当前登录用户绑定的 employee_id`。
- **工位 / 设备的部门过滤**：这两张表本身没有 `department_id`，dept_manager 的"本部门"过滤需通过当前占用/使用人关联部门，即 `JOIN employees e ON e.id = current_employee_id WHERE e.department_id IN (...)`。**闲置工位 / 在库设备**（`current_employee_id` 为 NULL）不归属任何部门，dept_manager **不可见**（由 IT/HR 全局管理）。
- **报表**：dept_manager 的报表同样按上述规则限定在本部门范围内聚合。

> 实现要点：登录用户通过 `users.employee_id` 找到员工记录，再定位其 `department_id`，这是数据权限的地基（见 §5.1、§5.5）。数据范围条件由 `core/deps.py` 统一产出并注入 repository 查询（见 §11）。

### 3.4 权限点清单（种子数据）

以下为系统内置权限点 `code`（由 `scripts/seed.py` 初始化写入 `permissions` 表），§3.2 权限矩阵与 §8.6 接口清单均引用这些 code：

| 模块 | 权限点 code |
|---|---|
| 部门 | `department:view` `department:create` `department:update` `department:delete` |
| 员工 | `employee:view` `employee:create` `employee:update` `employee:delete` `employee:resign` `employee:import` `employee:export` |
| 工位 | `workstation:view` `workstation:manage` |
| 设备 | `device:view` `device:manage` |
| 报表 | `report:view` |
| 系统 | `system:user` `system:role` `system:log` `system:dict` |

> 约定：工位/设备的写操作（分配/释放/领用/退还/维修/报废/编辑）统一收敛为 `workstation:manage` / `device:manage`；部门/员工的写操作拆分为 `create/update/delete` 等独立 code。

## 4. 关系模型

```
部门(department) 1 ── N 员工(employee)
员工(employee)  N ── 1 部门
员工(employee)  N ── 1 直属上级(employee 自关联，需防环)
工位(workstation) 1 ── 1 员工（当前占用；历史进 workstation_assignments）
设备(device)     N ── 1 员工（一人可多设备；历史进 device_assignments）
账号(user)       1 ── 0..1 员工（自助/主管账号绑定员工；纯管理员账号可不绑定）
角色(role)       N ── N 权限(permission)（通过 role_permissions）
```

**约束补充：**
- 直属上级自关联，保存/更新时校验**不能形成环**（A→B→…→A）。
- 工位「当前占用」以工位表 `current_employee_id` 为唯一事实来源；历史表只追加不改。

## 5. 数据库设计

> 通用约定：所有表含 `id CHAR(36) PK`；核心业务表含审计字段 `created_by / updated_by / created_at / updated_at`，可软删的含 `deleted_at`（NULL=未删）。时间统一存 UTC。
>
> **软删除与唯一约束**：`username`、`employee_no`、`workstation.code`、`asset_code`、`serial_number` 等唯一业务键的表，均加一个生成列 `active_flag TINYINT AS (IF(deleted_at IS NULL, 1, NULL)) STORED`，唯一索引建在 `(业务键, active_flag)` 上——即**唯一性仅在"未删除"记录间生效**（已删行 `active_flag=NULL`，MySQL 允许多个 NULL 并存，故不冲突）。如此软删后原工号/资产编号可被复用，而活跃记录仍严格唯一。下文各表凡标 `UNIQUE` 的业务键均按此实现。

### 5.1 users（账号表）※新增

**用途**：系统登录账号。凡能登录系统的人各对应一条记录；通过 `employee_id` 与员工档案绑定，是功能权限与数据权限判定的入口。账号与员工分离——纯管理岗可以只有账号、不绑定员工。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | CHAR(36) | PK | 主键，UUIDv7（有序，写入友好） |
| username | VARCHAR(64) | UNIQUE, NOT NULL | 登录名，全系统唯一，登录凭据 |
| password_hash | VARCHAR(255) | NOT NULL | 密码密文，bcrypt/argon2 加盐哈希，**严禁存明文或 MD5** |
| role_id | CHAR(36) | FK→roles.id, NOT NULL | 所属角色，决定功能权限与数据范围 |
| employee_id | CHAR(36) | FK→employees.id, NULL | 绑定的员工。dept_manager/employee 靠它定位「本部门/本人」；纯管理员账号可为空 |
| status | TINYINT | NOT NULL, default 1 | 账号状态：1=启用，0=禁用（禁用后不可登录，员工离职时置 0） |
| last_login_at | DATETIME(3) | NULL | 最近一次登录成功时间，用于安全审计 |
| pwd_updated_at | DATETIME(3) | NULL | 密码最后修改时间，可用于「密码过期强制改密」策略 |
| created_at | DATETIME(3) | NOT NULL | 创建时间（UTC） |
| updated_at | DATETIME(3) | NOT NULL | 最后更新时间（UTC） |
| deleted_at | DATETIME(3) | NULL | 软删除标记：NULL=有效，非空=已删除 |

索引：`uk_username(username)`、`idx_employee_id`。

### 5.2 roles（角色表）※新增

**用途**：定义系统中的角色（如超级管理员、HR、IT 等）。角色是「功能权限」与「数据范围」的载体，账号通过 `role_id` 关联角色获得权限。预置 5 个内置角色，同时允许后台新增自定义角色。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | CHAR(36) | PK | 主键，UUIDv7 |
| code | VARCHAR(32) | UNIQUE, NOT NULL | 角色编码，程序内引用用（如 `super_admin`、`hr_admin`），唯一且不建议修改 |
| name | VARCHAR(64) | NOT NULL | 角色显示名称（如「HR 管理员」） |
| is_builtin | TINYINT | NOT NULL, default 0 | 是否内置：1=系统内置（不可删除/改编码），0=自定义 |
| data_scope | VARCHAR(16) | NOT NULL | 数据范围：`all`=全局，`dept`=本部门(含子部门)，`self`=仅本人 |
| status | TINYINT | NOT NULL, default 1 | 状态：1=启用，0=禁用 |
| created_at | DATETIME(3) | NOT NULL | 创建时间 |
| updated_at | DATETIME(3) | NOT NULL | 更新时间 |

索引：`uk_code(code)`。

### 5.3 permissions / role_permissions（权限表 / 角色权限关联表）※新增

**用途**：`permissions` 是系统所有「功能权限点」的字典（如「设备管理」「查看日志」）；`role_permissions` 把权限点分配给角色，二者构成「角色—权限」多对多关系，即 §3.2 权限矩阵的落库形式。鉴权时：账号 → 角色 → 权限点集合。

**permissions（权限点字典）**

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | CHAR(36) | PK | 主键，UUIDv7 |
| code | VARCHAR(64) | UNIQUE, NOT NULL | 权限编码，格式 `模块:操作`（如 `device:manage`、`system:log`），接口鉴权按它匹配 |
| name | VARCHAR(64) | NOT NULL | 权限显示名（如「设备管理」） |
| module | VARCHAR(32) | NOT NULL | 所属模块（department/employee/workstation/device/report/system），便于分组展示 |

**role_permissions（角色-权限关联）**

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| role_id | CHAR(36) | FK→roles.id, NOT NULL | 角色 |
| permission_id | CHAR(36) | FK→permissions.id, NOT NULL | 权限点 |
| | | PK(role_id, permission_id) | 联合主键，保证同一角色不重复授予同一权限 |

### 5.4 departments（部门表）

**用途**：存储公司组织架构，通过 `parent_id` 自关联形成部门树。员工挂靠部门，dept_manager 的数据范围也由部门树推导。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | CHAR(36) | PK | 主键，UUIDv7 |
| name | VARCHAR(64) | NOT NULL | 部门名称 |
| parent_id | CHAR(36) | FK→departments.id, NULL | 上级部门；根部门为 NULL。递归可得完整部门树 |
| manager_id | CHAR(36) | FK→employees.id, NULL | **新增**：部门主管（员工）。用于识别 dept_manager，可为空 |
| sort_order | INT | NOT NULL, default 0 | 同级排序序号，越小越靠前 |
| status | TINYINT | NOT NULL, default 1 | 状态：1=启用，0=禁用 |
| created_by | CHAR(36) | NULL | 创建人账号 id（审计） |
| updated_by | CHAR(36) | NULL | 最后修改人账号 id（审计） |
| created_at | DATETIME(3) | NOT NULL | 创建时间 |
| updated_at | DATETIME(3) | NOT NULL | 更新时间 |
| deleted_at | DATETIME(3) | NULL | 软删除标记：NULL=有效 |

索引：`idx_parent_id`。

### 5.5 employees（员工表）

**用途**：员工人事档案主表，是工位、设备归属的核心主体。通过 `department_id` 挂靠部门、`direct_supervisor_id` 自关联表示汇报关系。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | CHAR(36) | PK | 主键，UUIDv7 |
| employee_no | VARCHAR(32) | UNIQUE, NOT NULL | 工号，全公司唯一，不可为空 |
| name | VARCHAR(64) | NOT NULL | 姓名 |
| gender | TINYINT | NOT NULL | 性别，取字典 `gender`（如 1=男，2=女，0=未知） |
| email | VARCHAR(128) | NULL, UNIQUE | 邮箱，可空；填了则不可重复（MySQL 唯一索引允许多个 NULL） |
| phone | VARCHAR(20) | NULL, UNIQUE | 手机号，可空；填了则不可重复 |
| department_id | CHAR(36) | FK→departments.id, NULL | 所属部门；未分配时可空 |
| direct_supervisor_id | CHAR(36) | FK→employees.id, NULL | 直属上级（自关联）；顶层可空。保存时校验**不得形成环** |
| position | VARCHAR(64) | NULL | 职位名称 |
| hire_date | DATE | NULL | 入职日期 |
| resign_date | DATE | NULL | **新增**：离职日期；在职时为空，离职处理时写入 |
| status | TINYINT | NOT NULL, default 1 | 在职状态：1=在职，0=离职 |
| created_by | CHAR(36) | NULL | 创建人账号 id（审计） |
| updated_by | CHAR(36) | NULL | 最后修改人账号 id（审计） |
| created_at | DATETIME(3) | NOT NULL | 创建时间 |
| updated_at | DATETIME(3) | NOT NULL | 更新时间 |
| deleted_at | DATETIME(3) | NULL | 软删除标记：NULL=有效 |

索引：`uk_employee_no`、`idx_department_id`、`idx_direct_supervisor_id`。

### 5.6 workstations（工位表）

**用途**：管理物理工位及其占用状态。`current_employee_id` + `status` 表达「当前谁在用」，是工位占用的唯一事实来源；历史变更进 §5.8。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|

| code | VARCHAR(32) | UNIQUE, NOT NULL | 工位编号（如 A-12-03），全局唯一 |
| location | VARCHAR(128) | NULL | 具体位置描述（楼层/区域） |
| type | VARCHAR(16) | NOT NULL | 工位类型，取字典 `workstation_type`（固定/共享/临时） |
| status | TINYINT | NOT NULL, default 1 | 状态：1=闲置，2=占用，3=预留，4=禁用 |
| current_employee_id | CHAR(36) | FK→employees.id, NULL | 当前占用员工；闲置/预留/禁用时为 NULL |
| assign_date | DATE | NULL | 当前占用的分配日期；未占用时为空 |
| version | INT | NOT NULL, default 0 | **乐观锁版本号**，每次更新 +1，防并发重复分配（详见 §10） |
| notes | VARCHAR(255) | NULL | 备注 |
| created_by | CHAR(36) | NULL | 创建人账号 id（审计） |
| updated_by | CHAR(36) | NULL | 最后修改人账号 id（审计） |
| created_at | DATETIME(3) | NOT NULL | 创建时间 |

| deleted_at | DATETIME(3) | NULL | 软删除标记：NULL=有效 |

### 5.7 devices（设备表）

### 5.8 workstation_assignments（工位分配历史表）

**用途**：记录每一次工位占用的完整历史（谁、从何时到何时、谁操作的）。主表只保当前状态，历史查询与审计靠这张流水表。一条记录 = 一段占用期。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | CHAR(36) | PK | 主键，UUIDv7 |

## 6. 功能模块详细设计

### 6.1 部门
- 基础 CRUD。**删除前置校验**：① 无子部门；② 无在职员工。任一不满足则拒绝（软删除）。
- 树形结构查询：返回带层级的部门树；提供「获取子部门ID集合」内部接口供数据权限使用。

### 6.2 员工
- 分页查询：按工号/姓名/部门/状态筛选，姓名/工号模糊匹配。
- 详情查看/编辑：含部门、直属上级、职位等；编辑上级时校验防环。
- 批量修改部门；**批量导入**（Excel 模板校验，异步 Celery）；导出 Excel（大数据量异步）。
- **离职处理（事务）**：`status=0`、写 `resign_date` → 自动释放占用工位（置闲置 + 写历史 end_date）→ 退还全部设备（置在库 + 写历史 return_date）→ **禁用其绑定的登录账号** → 记操作日志。

### 6.3 工位
- 列表：编号、位置、状态、占用员工。
- **分配（事务 + 乐观锁）**：`UPDATE ... SET current_employee_id=?, status=2, version=version+1 WHERE id=? AND status=1 AND version=?`；影响行数=0 说明已被占用，返回冲突。同时写历史。
- 释放：单个/批量，置闲置、写历史 end_date。
- 维护：设预留/禁用（装修维修场景）。

### 6.4 设备
- 列表：按资产编号/类型/状态/使用人筛选。
- 入库登记：新设备 status=在库。
- **领用/退还（事务 + 乐观锁）**：领用置使用中、写历史 checkout；退还置在库、`current_employee_id=NULL`、写历史 return。
- 维修/报废：置对应状态，报废/维修中**不可再分配**（分配接口校验 status）。
- 历史追溯：查某设备被哪些员工使用过。

### 6.5 报表
- 员工资产概览：某员工当前工位 + 持有设备。
- 闲置资产报表：闲置工位数、在库设备数。
- 部门设备统计：按部门汇总设备数量。
- **到期提醒**：Celery 定时扫描 `warranty_expire` 临近的设备，产出预警列表（可扩展邮件通知）。
- 报表建议增加**时间维度**（月度新增/退还趋势）。

## 7. 系统管理
- 账号管理：CRUD、启用/禁用、重置密码、绑定/解绑员工。
- 角色权限：角色管理 + 权限分配（勾选权限矩阵）。

## 8. API 设计与接口清单

### 8.1 统一响应体
```json
{ "code": 0, "message": "ok", "data": {}, "trace_id": "..." }
```
- `code=0` 成功；非 0 为业务错误码。HTTP 状态码正常使用（4xx/5xx）。

### 8.2 分页约定
- 入参：`page`(从1) 、`page_size`(默认20，上限100)、`keyword`、`sort`(排序字段)、`order`(asc/desc，默认 desc)、各筛选字段。
- 返回：`{ items:[], total, page, page_size }`。

### 8.3 错误码分段（示例）
| 区间 | 含义 |
|---|---|
| 1xxx | 认证/授权（如 1001 未登录、1003 无权限） |
| 2xxx | 参数校验 |
| 3xxx | 业务规则（如 3001 工位已被占用、3002 部门下有在职员工） |
| 5xxx | 系统内部错误 |

### 8.4 版本与文档
- 前缀 `/api/v1`；FastAPI 自动生成 Swagger（`/docs`）与 OpenAPI schema。

### 8.5 接口通用约定

- **Base URL**：所有接口以 `/api/v1` 为前缀，下表路径均省略该前缀。
- **认证**：除 `POST /auth/login`、`POST /auth/refresh` 外，全部接口需请求头 `Authorization: Bearer <access_token>`。
- **权限列**：标注调用该接口所需的权限点（对应 §3.2 权限矩阵 / §5.3 权限表）；`登录` 表示仅需登录态、无需特定权限（如本人改密）；`公开` 表示无需认证。
- **数据范围**：在功能权限之上，后端按当前账号角色的 `data_scope` **自动叠加行级过滤**（本部门/本人），无需前端传参。
- **列表接口**统一遵循 §8.2 分页与筛选；**写操作**统一记入操作日志（§5.10）。
- **命名规范**：资源用复数名词、小写中划线；集合上的动作用子路径（如 `/{id}/assign`）；筛选用查询参数。
- **参数校验**：由 Pydantic schema 统一约束——邮箱/手机号格式、枚举取值、字段长度、必填项；密码需满足复杂度（≥8 位且含大小写字母与数字），校验失败返回 2xxx。
- **健康检查**：另提供 `GET /health`（不带 `/api/v1` 前缀，**公开**），返回服务及依赖（DB/Redis）状态，供容器编排与 Nginx 上游探活。

> 说明：§3.2 中工位/设备的写操作（分配/领用/退还等）在接口层收敛为 `workstation:manage` / `device:manage` 两个权限点。

## 9. 认证与安全

- **认证（AuthN）**：除登录/刷新外全部接口需 JWT。采用 **access token（短，15min）+ refresh token（长，7天）** 双令牌。
- **授权（AuthZ）**：FastAPI 依赖注入统一鉴权——校验 token 有效 → 加载角色权限 → 校验接口所需 `权限code` → 注入数据范围过滤条件。功能权限与数据权限**双重校验**。
- **登出/失效**：JWT 无状态，登出/改密时将 jti 写入 **Redis 黑名单**（TTL=token 剩余有效期）。
- **密码与登录**：bcrypt 或 argon2 哈希加盐，禁止明文/MD5；登录失败达 **5 次锁定账号 15 分钟**（Redis 计数，阈值可配置）。
- **Redis 用途**：Token 黑名单、数据字典缓存、部门树/数据范围缓存、接口限流、分布式锁（分配类操作兜底）。
- **传输与配置**：全站 HTTPS；CORS 白名单；密钥/DB 口令走 `.env` + 环境变量，禁止入库入库入代码库。
- ORM 参数化查询防注入；关键写操作全部落操作日志。

## 10. 并发与一致性

- 分配/释放/领用/退还/离职等**跨表写操作全部包裹在数据库事务**中，主表与历史表同事务提交。
- 工位、设备主表加 `version` 乐观锁，配合条件更新（`WHERE status=可用 AND version=?`）防止并发双写。
- 「当前占用/使用」以主表 `current_employee_id` 为唯一事实来源，历史表**只追加不修改**。
- 高并发场景可叠加 Redis 分布式锁（按资源ID加锁）作为兜底。

## 11. 项目结构（后端）

采用**四层分层架构**，调用链自外向内为 `api`（装配/入口）→ `router`（控制器）→ `service`（业务逻辑）→ `repository`（数据访问）→ `model`（ORM 实体）。每层**单向依赖下一层**：router 不写业务、service 不碰 ORM 查询细节、repository 不含业务判断，便于替换实现与单元测试。`core` 承载配置与基础设施，`schemas` 定义出入参。前后端分离，前端 Vue 工程独立目录，本节聚焦后端。

```text

│   │   ├── config.py               # Pydantic Settings，读取 .env（DB/Redis/JWT 密钥等）
│   │   ├── database.py             # SQLAlchemy engine + Session 工厂
│   │   ├── redis.py                # Redis 连接池（Token 黑名单/缓存/分布式锁）
│   │   ├── security.py             # 密码哈希(bcrypt)、JWT 签发/校验
```

**四层职责与关键落点：**

| 层 | 目录 | 职责 | 关键设计落点 |
|---|---|---|---|
| ① api 装配层 | `api/` | 版本化主路由、聚合挂载各 router、全局前缀/依赖 | `/api/v1` 前缀、Swagger、全局异常处理 |
| ② router 控制器层 | `routers/` | 参数校验(schema)、鉴权(deps)、调 service、统一响应封装 | 权限点校验 + 数据范围条件注入（§8.6） |
| ③ service 业务层 | `services/` | 业务规则、**事务边界**、跨表级联；仅依赖 repository | 离职级联、分配/领用事务编排 |
| ④ repository 数据层 | `repositories/` | 封装 ORM CRUD/查询：分页、软删过滤、**乐观锁条件更新** | `assign_if_free()` 条件 UPDATE 返回受影响行数 |
| model 模型层 | `models/` | ORM 表映射、通用字段 Mixin | 软删/审计/`version` 列统一在 `base.py` |
| 基础设施 | `core/` | 配置、DB/Redis、安全、依赖注入 | 鉴权 + 数据范围条件产出集中在 `core/deps.py` |

**一次请求的流转（以「工位分配」为例）：**
`routers/workstations.py` 校验入参并校验 `workstation:manage` 权限 → 调 `services/workstation_service.assign()` **开启事务** → 调 `repositories/workstation_repo.assign_if_free(id, employee_id, expected_version)` 执行乐观锁条件更新（受影响行数=0 → service 抛错误码 3001）→ 同事务内 `assignment_repo.create()` 写分配历史 → 提交事务 → 控制器用统一响应体返回。

> 前端（Vite+Vue3）独立目录 `frontend/`，通过 Nginx 将 `/api` 反向代理到后端，`/` 指向前端打包产物（见 §12 部署）。

## 12. 工程化与部署

- **目录/迁移**：Alembic 管理表结构版本；禁止手工改线上表。注意 `departments.manager_id` ↔ `employees.department_id` 为**互相引用的外键**，需分两步迁移：先建两表（暂不含这对 FK），再 `ALTER TABLE ADD CONSTRAINT` 补外键，规避建表时的循环依赖。
- **环境**：dev / test / prod 三套配置，均用 `.env`。
- **容器化**：`docker-compose` 编排 api / mysql / redis / nginx / celery-worker / celery-beat。
- **Nginx**：反向代理 `/api` → FastAPI，静态资源托管前端打包产物。
- **测试**：pytest 覆盖核心业务（离职联动、分配并发、权限过滤）。
- **日志/监控**：结构化日志 + trace_id 贯穿；预留 Prometheus 指标位。
- **备份**：MySQL 定时全量 + binlog 增量备份。
- **日志治理**：`operation_logs` 按月分区或定期归档冷存，避免单表无限膨胀。
- **CI/CD**：lint（ruff）+ 测试 + 构建镜像 + 部署，建议接入流水线。
